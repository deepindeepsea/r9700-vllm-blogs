# R9700 vLLM MoE Concurrency Benchmark — 2026-05-28

**Hardware:** AMD Radeon AI PRO R9700, 32 GB GDDR6 (256-bit, ~640 GB/s), gfx1201 (RDNA4)
**Host:** the test host server, EPYC 9684X, Ubuntu 24.04, ROCm 7.2.1
**Stack:** `rocm/vllm-dev:rocm7.2.1_navi_ubuntu24.04_py3.12_pytorch_2.9_vllm_0.16.0`
**Model:** `cyankiwi/Qwen3-30B-A3B-Instruct-2507-AWQ-4bit` (30B total / **3B active**, compressed-tensors AWQ-INT4)
**Kernel:** `CompressedTensorsWNA16MoEMethod` on Triton attention; `VLLM_ROCM_USE_AITER=0` (required on RDNA4)
**Settings:** `--max-model-len 32768 --gpu-memory-utilization 0.92 --enable-prefix-caching --enable-auto-tool-choice`
**Weights on GPU:** 17.54 GiB · **KV cache pool:** 7.71 GiB → **84,224 tokens** (~20 simultaneous 4k-context users)
**Workload:** N concurrent streaming chat completions, distinct prompts (no prefix cache hit), 512 output tokens each, temperature 0.

## Headline finding

**Qwen3-30B-A3B (MoE) on a single R9700 delivers 6.7× the single-stream throughput of dense Qwen2.5-14B at the same VRAM footprint** — with a *larger* underlying model. At 16 concurrent users, the card serves **381 aggregate tok/s** (~24 tok/s per user, 142 ms first-token latency).

## Why MoE wins so big on this card

The R9700 at batch=1 is GDDR6-bandwidth-bound, not compute-bound (proven earlier this session: halving model size doubled tok/s). MoE breaks the link:

- **Dense decode:** every step fetches all 14B weights from GDDR6 — ~9.5 GiB at INT4 — for *every* output token.
- **MoE decode:** every step only fetches the active 3B expert weights — ~1.5 GiB at INT4 — *plus* shared attention weights. The remaining 27B of expert weights sit idle in VRAM until routed to.

So MoE essentially turns a 30B model into "3B-active speed, 30B-capacity quality." The compute headroom of the card finally gets used because we're no longer waiting on HBM/GDDR.

## Scaling shape

| N | Aggregate | Aggregate scaling factor | Per-user retention |
|---|---|---|---|
| 1  |  50.9 | 1.00× | 100% |
| 2  |  79.7 | 1.57× |  78% |

| 8  | 226.0 | 4.44× |  56% |
| 16 | 381.3 | 7.49× |  47% |

Unlike dense models (where concurrency scaling was perfectly linear in our prior 14B run), **MoE scaling is sub-linear** because concurrent users hit *different* experts, increasing aggregate bandwidth demand. This is the expected MoE trade — the per-user cost goes up with N, but starts from such a high baseline (51 tok/s) that it doesn't matter for typical agentic workloads.

## Operating ranges for agentic AI

For a small/medium agent fleet (your "3–4 active users" target):
- **N=1–4** is the comfortable single-user / paired-agent regime: 34–51 tok/s per user, TTFT ≤ 120 ms, < 280 W. Feels instant.
- **N=8** is the busy fleet regime: 28 tok/s per user (still very usable for tool-loops), 142 ms TTFT, 305 W. Headroom on temp and KV cache.
- **N=16** is the upper bound the card runs cleanly: 24 tok/s per user, 160 ms TTFT, 376 W peak power. KV cache still under-utilised (16 × 4k context = 64k tokens, ≪ 84k pool).

KV cache, not bandwidth, will eventually backpressure — likely around N=20–24 at 4k context, or N=10–12 at 8k context.

## What this means for the stack going forward

1. **Adopt Qwen3-30B-A3B-AWQ as the default served model.** It dominates dense 14B on every metric — per-user latency, aggregate throughput, TTFT, and even idle power — while delivering a stronger underlying model.
2. **Don't tune single-stream further.** We've gone from 7.6 → 51 tok/s by switching to MoE; remaining knobs (`--enforce-eager`, scheduler tweaks) move single-digit %.
3. **The fused-MoE config warning** ("Config file not found at .../E=128,N=768,device_name=AMD_Radeon_Graphics,dtype=int4_w4a16.json") suggests vLLM is using a default kernel config rather than a tuned one. **Tuning this could yield another 10–25% on aggregate throughput** at higher N. Worth a follow-up.
4. **Thermal/power headroom remains** at 376 W max vs ~330 W TBP (running over briefly OK), 90 °C max vs ~110 °C throttle. Card is the right size for this workload.
5. **Coder variant** (`cyankiwi/Qwen3-Coder-30B-A3B-Instruct-AWQ-4bit`) is the obvious next experiment if tool-calling becomes the hot path — same architecture, same speed, optimized for code/tool use.

## Addendum (2026-05-28, later): Attempted kernel tuning — negative result

Tried to close the "Using default MoE config" warning. Three approaches failed:

1. **`benchmark_moe.py --dtype int8_w8a16`** — int8 path requires per-token activation quantization which the harness doesn't enable (hardcoded assertion in `_int8_quantize`).
2. **`benchmark_moe.py --dtype int4_w4a16`** — not a CLI option in vLLM 0.16. Runtime kernel supports it; tuner doesn't.
3. **Borrow MI308X tuned config**, rename to `device_name=AMD_Radeon_Graphics,dtype=int4_w4a16.json`, patch in `SPLIT_K: 1`:
   - Config loaded cleanly (warning replaced by "Using configuration from...").
   - N=1: 49.9 tok/s vs 50.9 baseline (~2% regression, noise).
   - **N=16: 82.8 tok/s aggregate vs 381 baseline — 4.6x REGRESSION** (per-user 5.2 vs 24 tok/s).
   - CDNA3 tile choices (BLOCK=16/32/64, num_warps=2, matrix_instr_nonkdim=16, kpack=1) are tuned for 64-wide CDNA wavefronts, large LDS, and MFMA — none of which match RDNA4's 32/64-wide WGPs, smaller LDS, and WMMA. At concurrency the mistuned tiles spill or under-utilize.

**Conclusion:** the warning is cosmetic. AITER (AMD's blessed MoE kernel library, `VLLM_ROCM_USE_AITER=1`) is the real performance unlock — but it's CDNA-only today. RDNA4 is on the generic Triton fallback path until either upstream vLLM adds int4_w4a16 to `benchmark_moe.py` or AMD extends AITER to consumer/workstation RDNA. Reverted to default config; server back to baseline.

## Reproduce
- Bench script: `./bench_vllm.py` on the test host (now with modulo prompt indexing for N > 8).
- Launch:
  ```
  docker run -d --name vllm-qwen3-moe \
    --device=/dev/kfd --device=/dev/dri/renderD128 \
    --group-add video --group-add 109 --ipc=host --shm-size=16g \
    --security-opt seccomp=unconfined --cap-add=SYS_PTRACE \
    --network=host --restart unless-stopped \
    -v <hf-cache-dir>:/root/.cache/huggingface \
    -e HF_TOKEN=<your-token> -e CUDA_VISIBLE_DEVICES=0 -e VLLM_ROCM_USE_AITER=0 \
    -e HIP_FORCE_DEV_KERNARG=1 -e TORCH_BLAS_PREFER_HIPBLASLT=1 \
    -e PYTORCH_ALLOC_CONF=expandable_segments:True \
    rocm/vllm-dev:rocm7.2.1_navi_ubuntu24.04_py3.12_pytorch_2.9_vllm_0.16.0 \
    vllm serve cyankiwi/Qwen3-30B-A3B-Instruct-2507-AWQ-4bit \
      --dtype float16 --max-model-len 32768 --gpu-memory-utilization 0.92 \
      --enable-prefix-caching --enable-auto-tool-choice --tool-call-parser hermes \
      --served-model-name qwen3-moe --host 0.0.0.0 --port 8000
  ```
- Sweep: `python3 -u bench_vllm.py --levels 1,2,4,8,16 --out bench_moe_results.json`
