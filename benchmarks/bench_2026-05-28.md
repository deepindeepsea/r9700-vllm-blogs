# R9700 vLLM Concurrency Benchmark — 2026-05-28

**Hardware:** AMD Radeon AI PRO R9700, 32 GB VRAM, gfx1201 (RDNA4)
**Host:** the test host server, EPYC 9684X, Ubuntu 24.04, ROCm 7.2.1
**Stack:** `rocm/vllm-dev:rocm7.2.1_navi_ubuntu24.04_py3.12_pytorch_2.9_vllm_0.16.0`
**Model:** `Qwen/Qwen2.5-14B-Instruct-AWQ` (AWQ-INT4, Triton kernel)
**Settings:** `--max-model-len 32768 --gpu-memory-utilization 0.92 --enable-prefix-caching`,
`VLLM_ROCM_USE_AITER=0` (required on RDNA4)
**Workload:** N concurrent streaming chat completions, distinct prompts (no prefix cache hit),
512 output tokens each, temperature 0.

## Key findings

1. **Linear throughput scaling through N=4.** Each additional concurrent user adds ~7.5 aggregate tok/s with zero degradation to existing users. vLLM continuous batching is fully exploiting the card.
2. **Single-stream is the bottleneck, not multi-user.** The 7.5 tok/s ceiling per request reflects kernel/memory-bandwidth limits on the AWQ Triton path for gfx1201 — not contention from other users.
3. **TTFT is fast** (~200 ms steady) and improves slightly with N due to warm CUDA graphs.
4. **Thermal/power headroom exists**: 318 W max vs ~330 W TBP; 85 °C max vs ~110 °C throttle.
5. **KV cache pool barely touched.** Single 512-token completion uses < 1% of the ~19 GiB pool, so estimated ceiling before backpressure is ~16–30 simultaneous agent sessions at 4k–8k context.

## What this implies for optimization

| Goal | What to change |
|------|----------------|
| Lower per-stream latency (single user feels faster) | Try GPTQ-INT4 (Marlin path), build vLLM from `main`, watch ROCm for AITER RDNA4 enablement |
| Higher aggregate throughput | Already near-linear; only worth tuning if you push past N=8–16 |
| Lower idle VRAM | Drop `--gpu-memory-utilization` to 0.75 (still fits ~80k tokens of KV cache) |
| Lower first-token latency for repeated agent calls | Already on; `prefix_cache_hit_rate` improves when same system prompt repeats |
