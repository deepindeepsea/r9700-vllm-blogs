Part 13 · The 35B only one runtime could load
Qwen3.6-35B-A3B · qwen3_5_moe · hybrid linear-attention

A 35B MoE landed on a single 32 GB card — and only one runtime could even open the file.

The second baseline was supposed to be a routine "bigger model, slower numbers" run. Instead it turned into a loadability story: llama.cpp's converter recognized a brand‑new hybrid‑attention architecture and benchmarked it the same afternoon — while vLLM on the ROCm‑navi container flatly refused to register the arch at all. This is the number that exists because of the runtime that could load it.

The headline. On one R9700, Qwen3.6‑35B‑A3B (Q4_K_M, 20 GB) sustained 2,752 tok/s prefill and 78.5 tok/s decode — and lost only ~8% prefill across an 8× jump in context. That flat curve is the whole point.
Single AMD Radeon AI PRO R9700 (gfx1201, RDNA4, 32 GB) · ROCm 7.2.1 · llama.cpp build 764f1e6 · -fa 1
Meet the architecture

Not just bigger — structurally different.

Qwen3.6‑35B‑A3B is a multimodal, hybrid linear‑attention MoE (Qwen3_5MoeForConditionalGeneration, model_type qwen3_5_moe). Its text trunk mixes full attention with linear‑attention blocks — the same family idea behind gated‑DeltaNet. That hybrid trunk is exactly why its prefill curve behaves so differently from a plain MoE.

40Layers

Hidden size 2048, head_dim 256, 16 query heads / 2 KV heads. Full‑attention interval mixed with linear‑attention blocks.

256Experts / 8 active

moe_intermediate 512. A far wider expert pool than the 30B's 128‑expert layout — more routing, more weight to stream.

248kVocabulary

248,320 tokens. A heavy embedding + LM head that the 30B (much smaller vocab) simply doesn't pay for.

4.88Bits / weight (Q4_K_M)

20 GB on disk — a comfortable fit inside the R9700's 32 GB, with headroom for KV cache and activations.

The plot twist

Two runtimes, one file — only one opened it.

Before a single token was measured, the model had to load. This is where the two paths diverged hard.

The finding that matters

An 8× longer prompt cost only ~8% throughput.

Prefill on a standard attention stack scales badly with context — the attention matrix grows quadratically, so throughput drops sharply as prompts get longer. Qwen3.6‑35B‑A3B barely flinched: from pp512 to pp4096 it shed only ~8%. That near‑flat line is the fingerprint of the hybrid linear‑attention trunk (the ssm_a / ssm_alpha state‑space blocks), where long‑context work stays close to linear instead of quadratic.

The curve

~8% decay / 8× context

pp512 → pp4096: 2,752 → 2,531. Almost flat where a quadratic model collapses.

The cause

Hybrid linear attention

Full‑attention blocks interleaved with linear/state‑space blocks keep the long‑context cost sub‑quadratic.

The consequence

Long prefill is the arena

If context is cheap to extend, the payoff from a purpose‑built long‑prefill kernel compounds — exactly the R9700 optimization target.

35B vs 30B — the honest comparison

What the extra size actually costs.

Against the Part 12 30B baseline (same card, same HIP path), the 35B pays a predictable tax — bigger vocab, 256‑expert routing, head_dim 256. No surprises, and no free lunch:

The 35B is slower at the short‑context peak — that's the cost of 256 experts, a 248k vocab and head_dim 256. But it gives back a flatter long‑context curve. If your workload is long prompts, the size tax shrinks as context grows.
Where this points

The flat curve is a kernel invitation.

A model whose long‑prefill cost is already sub‑quadratic is the ideal target for a hand‑written HIP kernel: the arithmetic is regular, the working set is bounded, and every microsecond saved on the linear‑attention scan is multiplied across long contexts. That is precisely the gated‑DeltaNet chunked‑scan kernel on the R9700 optimization runway — the RDNA4 answer to a q36‑class prefill engine.

Path A · prefer first

MXFP4 → INT4 (AWQ)

Requant experts offline, route through the wvSplitK INT4 MoE GEMM — vindicated on 30B (75 µs, no hang). The 35B hang is layout‑specific, not a kernel bug.

Path B

MXFP4 → FP8 (E4M3)

Upcast experts to FP8 to exploit RDNA4 FP8 WMMA — a second route onto the tensor units.

The new kernel

Gated-DeltaNet chunked scan

A HIP scan kernel for the linear‑attention trunk — where this model's flat curve turns kernel effort into throughput.

The through‑line. Part 12 proved the measurement discipline. Part 13 found the model whose architecture rewards the next step — and confirmed that on the R9700, the runtime that could load it is the one that will host that kernel.
The recipe

Reproduce the second baseline.

From public weights to GGUF to numbers — the exact path.

# 1. Convert the HF checkpoint to a text GGUF (converter registers the arch).
python convert_hf_to_gguf.py Qwen3.6-35B-A3B --outfile qwen36-35b-a3b-f16.gguf

# 2. Quantize to Q4_K_M (~20 GB, 4.88 bpw).
llama-quantize qwen36-35b-a3b-f16.gguf qwen36-35b-a3b-Q4_K_M.gguf Q4_K_M

# 3. Benchmark prefill (512/2048/4096) + decode, flash-attn on.
llama-bench -m qwen36-35b-a3b-Q4_K_M.gguf -ngl 999 -fa 1 -p 512,2048,4096 -n 128
# -> pp512 2752 / pp2048 2651 / pp4096 2531 / tg128 78.5 tok/s
